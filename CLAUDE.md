## Output a full Godot specification report BEFORE writing any port code

- **The very first step of any Godot â†’ Swift port task is to produce a written specification report covering the entire feature, and present it to the user. Do NOT begin editing Swift until the report exists.** The report is mandatory even if the change feels small (one stage tweak, one particle, one timing) â€” the surrounding graph is what gets dropped silently when this step is skipped.
- **Why:** repeatedly the port has been judged "done" against a partial mental model that omitted child nodes, sibling state-machine stages, signal listeners, animation tags, particle materials, audio cues, or auxiliary helpers (raycast / marker / laser). Recent example: LanceThrow shipped without the `DamageOnTouch2` long-throw hitbox, the `laser` aim overlay, the `hide_far_trails` distance gate, the firetip/evilfire emission stop on state 1â†’2, the screenshakes, and the audio cues â€” none of these were in the reasoning when Swift was written. Writing the spec FIRST forces the omissions to surface as explicit checklist rows instead of silent drops.
- **How to apply:** before the first Edit/Write on a port, output a markdown report with: (1) the full state machine of every controlling script (every stage, transition condition, side-effect â€” gravity, vy, screenshake, audio, animation, signal, tween_speed); (2) every child node in every relevant `.tscn` listed by name + type + position + key parameters (amount/lifetime/material/texture/loop) â€” including `DamageOnTouch*`, `Marker2D`, `RayCast2D`, `AudioStreamPlayer2D`, `GPUParticles2D`, secondary `AnimatedSprite2D`, `CanvasItemMaterial` overlays; (3) every signal / `Event.emit_signal` / `Event.listen` involved; (4) every particle process material (`*.tres`) with direction/spread/gravity/anim params; (5) animation tag names + frame ranges + `loop` flags; (6) auxiliary scripts the feature depends on (e.g. `Tools.timer_p`, `BossRNG`); (7) **a "Swift gap" table marking each row as âś… ported / âš ď¸Ź partial / âťŚ missing**, with the file:line of the Swift counterpart when present. Then ask the user which gaps to address before coding.
- **Scope:** every Godot port â€” Boss, Player, enemy, weapon, projectile, stage object, intro/outro sequence, ability, particle, shader, audio. The rule is "report first, code second" with no exceptions; partial-feature audits (e.g. "fix this one bug") still require the spec for the surrounding feature so the fix doesn't drop adjacent behavior.

## Visuals MUST come from the Godot sprite sheet

- **Render every game visual (projectiles, boss attacks, effects, characters) from the Aseprite sprite sheet in `Mega-Man-X8-16-bit/src/...` â€” do NOT reconstruct art from compound `SKNode` / flat-color `SKSpriteNode` primitives.**
- **Why:** the Godot atlases are the reference truth. Hand-rolled geometry drifts in color, shape, and animation from the original, and produces regressions that only surface during visual review. Example: OverdriveAttack's beam was re-created with red/orange additive rects before we checked `sigma_laser.png`, which is green with a rounded pill cap and skull-shaped muzzle â€” the drift was invisible to tests but obvious on screen.
- **How to apply:** before writing any visual, locate the matching Aseprite PNG + JSON under `Mega-Man-X8-16-bit/src/Actors/...`. Copy it to `Public/assets/sprites/<group>/<name>.{png,json}`, load it via `SpriteLoader.load(group:name:)` in `main.swift`, and bind it to the node via `attachAtlas(...)`. If the renderer appears to drop the sprite, fix the rendering path â€” do not replace the art with primitives.
- **Scope:** applies to every visual. No "temporary" primitive stand-ins; a missing sprite means the renderer needs fixing, not the art.

## Extract the FULL Godot context before porting ANY feature

- **Never port a Godot feature from a single file.** Before writing any Swift, walk the full neighborhood of every Godot script you intend to port: (1) the `extends` chain up to `BaseAbility` / `Actor` / `Node2D`, (2) every other script whose `_StartCondition` / `_EndCondition` / `_Interrupt` references this one's state, signals, or `class_name`, (3) every `Event.emit_signal` it emits and every `Event.listen` it subscribes to (and the listeners on the other side), (4) the matching Aseprite atlas tag(s) and frame ranges in `Mega-Man-X8-16-bit/src/.../*.json`, (5) any `@onready` particles / scenes / shaders the script attaches.
- **Why:** porting one Godot file in isolation drops mid-graph transitions, signal listeners, and animation tags silently. The single-file port reads as "complete" against its own E2E test but breaks the chain `X â†’ Y â†’ Z` that no single test exercises. The walljumpâ†’wallslide regression (after a walljump the player could not re-enter wallslide because `updateWallSlide` excluded `state == .wallJump`) is the canonical example, but the same trap applies to bosses, enemies, weapons, and stage objects.
- **How to apply:** for each Godot script you touch, write a checklist BEFORE editing Swift â€” list every script in (1)â€“(5) and the specific lines/tags you will mirror. Do not consider the port done until every checklist item has a corresponding line in the Swift port (or a comment explaining why it is intentionally omitted, e.g. dash variant deferred). When in doubt, add the missing transition rather than assume it doesn't matter.
- **Scope:** every Godot port â€” Player, Boss, enemies, weapons, stage objects, intro sequences. Mid-graph transitions and signal listeners are exactly where this rule pays off.

## Particles, trails, and overlay nodes are NOT optional decoration

- **Every `GPUParticles2D`, secondary `AnimatedSprite2D`, trail, and `CanvasItemMaterial`-based overlay node in a `.tscn` is part of the attack's visual signature â€” port all of them, not just the primary sprite + script.**
- **Why:** repeated regressions where the main sprite + state machine were ported correctly but particles / trails / shader overlays were silently dropped. Sigma's LanceThrow shipped as a single flying sprite with no trail or fire particles â€” the Godot original is a static spear at the wall hit-point with 4 stagger-faded trail sprites (`trail`, `trail2`, `trail3`, `trail4` using `sigma_trail.png`) plus `firetip` (`sigma_particles2.png` with 3Ă—3 anim atlas) plus `evilfire_particles` (`sigma_particles.png` drifting upward). The drop is invisible to logic tests because hitboxes / damage / state transitions still pass â€” only visual review catches it. Same pattern previously bit OverdriveAttack's beam (charge sparks + muzzle flash were initial misses).
- **How to apply:** when reading a `.tscn`, list **every** child node â€” not just the script-attached ones. For each `GPUParticles2D` / extra `AnimatedSprite2D` / `CanvasItemMaterial` overlay, decide before coding: (a) port it faithfully via `SKEmitterNode`-like spawning, additive sprite stagger, or `BossEffects` helpers, OR (b) explicitly comment in the Swift port why it is omitted. "I forgot" is not a valid (b). Copy the matching texture (`Mega-Man-X8-16-bit/src/Effects/Textures/*.png`) into `Public/assets/sprites/effects/` and load it via `SpriteLoader` â€” do not invent primitive stand-ins.
- **Scope:** every visual port. The presence of even one `GPUParticles2D` or secondary sprite node in the `.tscn` triggers this rule.
